**Key points:**

- Data is transmitted via HTTPS POST in 10-second batches

### 1.3 Does the endpoint executable require administrator privileges?

**No.** The tracker agent runs entirely in user space and does not require administrator/root privileges on any platform.

| Platform | Privilege Level | Details |
|----------|----------------|---------|
| **macOS** | Standard user | Uses AppKit (pyobjc) for window detection, pynput for input counting, Quartz for idle time. Runs as a LaunchAgent under the user's session. Requires Accessibility permission (System Preferences → Security & Privacy → Accessibility) — this is a user-level grant, not admin. |
| **Windows** | Standard user | Uses Win32 APIs (GetAsyncKeyState, GetCursorPos, GetLastInputInfo, GetForegroundWindow). No elevated privileges needed. Runs as a Task Scheduler entry under the user's logon session. |
| **Linux** | Standard user | Uses xdotool for window detection, pynput for input, xprintidle for idle time. |

**Network:** The agent only makes **outbound HTTPS POST** requests. It does not listen on any port, does not open any inbound connections, and does not require firewall rule changes.

The application supports multiple distribution methods:

| Method | Platform | Details |

### 1.5 Is the executable code signed? If yes, provide certificate details.

**Current state:** The PyInstaller-built executables are **not currently code-signed**. This is a known gap.

**Remediation plan:**
- **macOS:** Sign with an Apple Developer ID certificate using `codesign` and notarize with `xcrun notarytool`. This is required for Gatekeeper bypass without manual user override.
- **Windows:** Sign with an EV Code Signing certificate using `signtool.exe`. This eliminates SmartScreen warnings.

**Recommendation for security team:** Code signing should be implemented before enterprise-wide deployment. The build scripts (`installer/mac/build.py`, `installer/windows/build.py`) have been designed to accommodate signing as a post-build step.

## 2. Data Collection & Privacy

### 2.1 Provide the exact list of telemetry fields collected by the application

Each telemetry sample (collected once per second) contains exactly these fields:

| Field | Type | Description | Sensitive? |
|-------|------|-------------|------------|
| `timestamp` | ISO 8601 string | UTC time of the sample | No |
| `user_id` | String | Employee identifier (from config, not derived from system) | Low — configured by IT |
| `app_name` | String | Foreground application name (e.g., "Cursor", "Google Chrome") | Low |
| `window_title` | String | Window/tab title — **subject to privacy mode** (see 2.2) | Configurable |
| `keystroke_count` | Integer | Number of key presses in the interval (count only, **never content**) | No |
| `mouse_clicks` | Integer | Number of mouse clicks (count only, **never target/position**) | No |
| `mouse_distance` | Float | Total mouse cursor travel in pixels | No |
| `idle_seconds` | Float | OS-reported seconds since last user interaction | No |
| `distraction_visible` | Boolean | Whether a non-productive app was visible on another monitor | No |

**What is NOT in a sample:**
- No keystroke content, key identifiers, or key sequences
- No mouse click coordinates or targets
- No screenshot or screen pixel data
- No clipboard contents
- No file names, paths, or document contents
- No browsing URLs
- No network traffic or DNS logs

---

### 2.2 Confirm that the tool does NOT collect the following

| Data Type | Collected? | Verification |
|-----------|------------|--------------|
| **Clipboard contents** | **NO** | No clipboard API calls anywhere in the codebase. The tracker only calls: `get_active_window()`, `get_and_reset_counts()`, `get_idle_seconds()`, `get_visible_windows()`. |
| **Screen recordings** | **NO** | No screenshot, screen capture, or pixel-reading APIs. The tracker never imports imaging libraries. |
| **File names or document contents** | **NO** | No filesystem access, no `os.listdir`, no file reading. The only file the tracker writes is `buffer.json` (its own offline queue). |
| **Browser URLs** | **NO** | The tracker captures the **window title** only (e.g., "GitHub - Pull Request - Google Chrome"), not the URL bar. In `redacted` mode (default), even the title is reduced to a classification keyword (e.g., "youtube") or empty string. |
| **Chat content** | **NO** | Only the application name is captured (e.g., "Slack", "Microsoft Teams"). Message content is never accessible to the tracker. |

**Window title privacy modes** (configurable via `WINDOW_TITLE_MODE`):

| Mode | Example Title | Stored As |
|------|--------------|-----------|
| `redacted` **(default)** | "RE: Salary Review - Gmail" | `"gmail"` (keyword only) |
| `full` | "RE: Salary Review - Gmail" | `"RE: [REDACTED] - Gmail"` (emails/IDs scrubbed) |
| `off` | "RE: Salary Review - Gmail" | `""` (empty string) |

Even in `full` mode, automated regex scrubbing removes:
- Email addresses
- Numbers with 8+ digits
- ID patterns (e.g., `CA12345`, `TKT-2024001`)
- Custom patterns via `TITLE_SCRUB_PATTERNS` env var

### 2.3 Is any telemetry stored locally before being transmitted? If yes, for how long?

**Yes, in two scenarios:**

| Scenario | Local Storage | Duration | Location |
|----------|--------------|----------|----------|
| **Normal operation** | 10-second in-memory batch | ~10 seconds | RAM only (Python list) |
| **Backend unreachable** | `buffer.json` (JSON-lines file) | Until next successful connection (flushed on tracker restart) | `tracker/buffer.json` in the install directory |

**Details:**
- During normal operation, samples are held in memory for up to 10 seconds before being POSTed. On successful transmission, the in-memory batch is cleared.
- If the backend is unreachable (network down, server maintenance), events are appended to `buffer.json` in JSON-lines format.
- On next tracker startup, `buffer.json` is read, sent to the backend in 100-event chunks, and cleared.
- `buffer.json` is stored in the tracker's working directory — not in temp or system directories.
- There is no automatic TTL on `buffer.json`. It persists until successfully flushed or manually deleted.

**Recommendation:** For environments requiring strict local data policies, the buffer file path can be configured via `BUFFER_FILE` and pointed to an encrypted volume.

---

### 2.4 Are telemetry records anonymized or aggregated before being sent to the backend?

**No — raw, per-second samples are transmitted to the backend.** This is by design:

- **Why raw:** Classification accuracy depends on having the complete 60-second event sequence. Pre-aggregating on the client would lose the signal needed for confidence scoring, anti-cheat detection, and proportional app time splitting.
- **Privacy is enforced at the field level:** The tracker never collects sensitive content (keystroke content, URLs, clipboard, screenshots). What is transmitted (app name, counts, idle time) is inherently non-sensitive.
- **Window titles are privacy-processed before transmission:** In `redacted` mode (default), titles are reduced to classification keywords on the tracker before leaving the machine. In `full` mode, sensitive patterns are scrubbed before transmission.
- **Aggregation happens server-side at read time:** Raw events are grouped into 60-second buckets and classified only when a dashboard requests the data. The raw events themselves have a configurable retention period (default 14 days) and are auto-purged.

## 3. Data Transmission & API Security

### 3.1 How is telemetry transmitted to the backend API (protocol and encryption)?

| Aspect | Details |
|--------|---------|
| **Protocol** | HTTPS (TLS 1.2+) when using ngrok tunnel or a reverse proxy |
| **Transport** | HTTP POST with JSON body (`Content-Type: application/json`) |
| **Endpoint** | `POST /track` or `POST /tracker/ingest` |
| **Batch size** | ~10 events per request (configurable via `BATCH_INTERVAL_SEC`) |
| **Payload limit** | 512 KB max (configurable via `MAX_REQUEST_SIZE_KB`) |
| **Timeout** | 5-second client timeout per request |
| **Retry** | On failure, events are buffered locally and retried on next startup |

**In development:** HTTP to `localhost:5000` (TLS not applicable for loopback).
**In production:** HTTPS via ngrok (TLS 1.3), AWS ALB (TLS 1.2+), or any reverse proxy with TLS termination. The `BACKEND_URL` is configured at build time or in `.env`.

The Python `requests` library is used for transmission, which delegates TLS to the system's OpenSSL installation.

---

### 3.2 How does the endpoint agent authenticate to the backend API?

**Two-factor device authentication:**

| Header | Purpose |
|--------|---------|
| `Authorization: Bearer <device-token>` | A per-team device token issued by a manager via the admin dashboard. The token is SHA-256 hashed before storage — the plaintext is only shown once at creation time. Tokens have configurable expiry and can be revoked instantly. |
| `X-LAN-ID: <employee-lan-id>` | The employee's Windows LAN ID (or configured identifier). The backend validates that this LAN ID corresponds to a known `User` record with an active `Membership` in the team associated with the device token. |

**Server-side validation flow:**
1. Extract Bearer token from `Authorization` header
2. SHA-256 hash the token and look up `TrackerDeviceToken` in the database
3. Check: token exists, not expired, not revoked
4. Extract `X-LAN-ID` header
5. Look up `User` by `lan_id`, verify active `Membership` in the token's team
6. If any check fails → 401 Unauthorized or 403 Forbidden

**In demo mode** (`DEMO_MODE=true`): Authentication is bypassed. The `user_id` field in the telemetry event is used directly. **Demo mode must be disabled for production deployment.**

---

### 3.3 Is API hosted on KONG or AWS? Are API requests protected against replay attacks or misuse?

**Hosting:**
The API is currently a self-hosted Flask application. It can be deployed on any infrastructure:
- **AWS** (EC2, ECS, Fargate, Lambda) — see [DEPLOYMENT_AWS.md](DEPLOYMENT_AWS.md)
- **On-premises** behind a reverse proxy (nginx, HAProxy)
- **ngrok** for development/demo access

It is not currently behind a KONG API gateway, but can be placed behind one for additional rate limiting, logging, and API key management.

**Replay protection:**

| Control | Implementation |
|---------|---------------|
| **OIDC nonce + state** | SSO login flow uses cryptographic `nonce` (32 bytes, `secrets.token_urlsafe`) and `state` parameters to prevent authorization code replay and CSRF on the callback. |
| **Session binding** | Admin sessions are server-side (Flask-Session) with `HttpOnly`, `Secure`, `SameSite=Lax` cookies. Session ID is regenerated on login. |
| **Rate limiting** | `120/min` per device on ingest, `5/min` on login, `30/min` on admin mutations. Prevents brute-force and flood attacks. |
| **Device token expiry** | Tokens have configurable expiry and can be instantly revoked. |
| **Input validation** | Every telemetry event is validated via `validate_event()` — type checks, range checks, required fields. |
| **Payload size limit** | 512 KB max request body. Oversized requests return 413. |

**Not yet implemented:**
- Telemetry event deduplication (duplicate events from retried batches are stored twice; this has no impact on classification since events are deduplicated by timestamp during bucketing).
- Per-event cryptographic signatures or HMAC.

---

### 3.4 Are rate limiting and input validation implemented on the backend API?

**Yes, both are implemented.**

**Rate Limiting** (Flask-Limiter):

| Endpoint Class | Limit | Purpose |
|---------------|-------|---------|
| Tracker ingest (`/track`, `/tracker/ingest`) | 120 requests/minute per IP | Prevents flood from compromised agents |
| Admin login (`/admin/login`) | 5 requests/minute per IP | Prevents brute-force |
| Admin mutations (DELETE, POST) | 30 requests/minute per IP | Prevents mass data operations |

Rate limit exceeded → HTTP 429 with JSON error body.

**Input Validation** (`validate_event()` in `backend/utils.py`):

| Field | Validation |
|-------|-----------|
| Event body | Must be a JSON object |
| `timestamp` | Must be ISO 8601 string |
| `app_name` | Must be a string |
| `keystroke_count` | Must be integer ≥ 0 |
| `mouse_clicks` | Must be integer ≥ 0 |
| `mouse_distance` | Must be number ≥ 0 |
| `idle_seconds` | Must be number ≥ 0 |
| Request body | Max 512 KB (returns 413 if exceeded) |

Invalid events are rejected with HTTP 400 and a descriptive error message. Valid events in the same batch are still ingested (partial success).

| Layer | Encryption | Details |
|-------|-----------|---------|
| **PostgreSQL** | Depends on deployment | AWS RDS: AES-256 encryption at rest by default. Self-hosted: configure `data_encryption` in `postgresql.conf` or use LUKS/dm-crypt at the volume level. |

### 4.3 What is the retention period for telemetry data?

| Setting | Default | Configurable? |
|---------|---------|--------------|
| `DATA_RETENTION_DAYS` | **14 days** | Yes, via `.env` |

**Implementation:**
- On every backend startup, the system automatically deletes telemetry events older than `DATA_RETENTION_DAYS`.
- A manual cleanup endpoint (`POST /cleanup`) is also available.
- Cleanup is audit-logged: `retention_cleanup` action records the number of events deleted.

**Data access is limited to:**

**If export capability is needed in the future, the following controls should be implemented:**

- Watermarking with the exporting manager's identity

## 5. Access Control & Monitoring

### 5.1 How are users authenticated to the dashboard (SSO, MFA)?

| Dashboard | Authentication Method |
|-----------|---------------------|
| **Admin Dashboard** | **OIDC SSO** (Azure AD, Okta, or any OIDC-compliant IdP). Authorization Code Flow with PKCE. MFA is enforced by the IdP (not by Zinnia Axion directly — the IdP's MFA policy applies). |
| **User Dashboard** | Currently unauthenticated (public read-only endpoints). Designed for individual employee self-service access. |

**Admin SSO flow:**
1. Manager opens admin dashboard → redirected to IdP login page
2. Manager authenticates with company credentials + MFA (as enforced by IdP)
3. IdP returns authorization code → backend exchanges for tokens
4. ID token validated (signature via JWKS, issuer, audience, expiry, nonce)
5. Manager identity resolved from email claim → session created
6. Server-side session with `Secure`, `HttpOnly`, `SameSite=Lax` cookie, 8-hour lifetime

**Session hardening:**
- Session stored server-side (Flask-Session)
- Cookie flags: `Secure=True` (HTTPS only), `HttpOnly=True` (no JS access), `SameSite=Lax`
- Session lifetime: 8 hours (configurable via `PERMANENT_SESSION_LIFETIME`)
- Session cleared on logout

---

### 5.2 What role-based access controls (RBAC) exist?

| Role | Permissions | Scope |
|------|------------|-------|
| `user` | View own dashboard (user dashboard) | Own data only |
| `manager` | View team leaderboard, drill-down to user detail, delete user data, assign/remove users, manage device tokens | Own team + descendant teams only (hierarchical) |
| `superadmin` | All manager permissions across all teams | All teams |

**Enforcement:**
- Every admin endpoint is protected by `@admin_required` (checks role is `manager` or `superadmin`)
- Every data access is scoped by `@team_scoped` (computes the manager's team subtree)
- Every user-level operation validates via `assert_user_in_scope()` — the user's team must be within the manager's subtree
- Cross-team access attempts return HTTP 403 and are recorded in the audit log as `idor_user_blocked` or `idor_team_blocked`

**Team hierarchy isolation:**
- A manager at Lifecad sees Lifecad + its child team Axion — but NOT the parent Engineering team or sibling Fast team
- `team_id` is **never** accepted from client input — it is always derived from the authenticated manager's session

---

### 5.3 Are dashboard access logs and API logs retained?

**Yes.**

| Log Type | Storage | Retention | Details |
|----------|---------|-----------|---------|
| **Audit log** | `audit_log` database table | Indefinite (not subject to data retention policy) | Every admin action: login, logout, delete, assign, remove, move, token create/revoke, IDOR blocks |
| **Application logs** | stdout/stderr (configurable to file or log aggregator) | Depends on deployment | Flask/Werkzeug request logs, tracker ingest logs, error logs |
| **Request correlation** | `g.request_id` (UUID) | Per-request | Every request gets a UUID set in the `X-Request-Id` response header. Audit log entries include this ID for correlation. |

**Audit log fields per entry:**
`actor`, `action`, `target_user`, `ip_address`, `user_agent`, `request_id`, `actor_user_id`, `actor_team_id`, `metadata` (JSON), `created_at` (UTC)

---

### 5.4 Are alerts generated for abnormal activity (large exports, unusual access) and is this monitored?

**Current state:**
- **IDOR attempts are logged** in the audit table with actions `idor_user_blocked` and `idor_team_blocked`. These can be queried via `/admin/audit-log`.
- **Rate limit violations** return HTTP 429 and are logged by Flask-Limiter.
- **Bot-like input patterns** are detected and recorded in the bucket's `reason` field as `bot_like_input_pattern`.

**Not yet implemented:**
- Real-time alerting (email, Slack, PagerDuty) for IDOR attempts or rate limit violations
- Anomaly detection dashboards
- SIEM integration (Splunk, ELK)

**Recommendation:** For production, configure log forwarding to your organization's SIEM. The structured audit log (with `request_id`, actor identity, IP, and User-Agent) is designed for easy ingestion by log aggregators. Alert rules should be created for:
- `idor_user_blocked` / `idor_team_blocked` actions (possible insider threat)
- High-volume 429 responses (possible abuse)
- Login from unexpected IPs (requires SIEM correlation with IdP logs)

## Summary of Security Controls

| Control | Status | Notes |
|---------|--------|-------|
| OIDC SSO for admin | Implemented | Azure AD / Okta compatible |
| Device token auth for tracker | Implemented | SHA-256 hashed, expiry, revocable |
| RBAC (manager, superadmin) | Implemented | Enforced on every admin endpoint |
| Hierarchical team isolation | Implemented | Recursive CTE (Postgres) / BFS (SQLite) |
| IDOR prevention | Implemented | `assert_user_in_scope()`, audit-logged |
| Server-side sessions | Implemented | Secure, HttpOnly, SameSite=Lax |
| CSRF protection | Implemented | Flask-WTF |
| Rate limiting | Implemented | Per-endpoint class |
| Input validation | Implemented | Type + range checks on every event |
| Security headers | Implemented | HSTS, CSP, X-Frame-Options, etc. |
| Audit logging | Implemented | Immutable table, 12+ action types |
| Data retention | Implemented | Auto-purge, configurable (default 14d) |
| Anti-cheat detection | Implemented | Bot-like input, multi-monitor distraction |
| Window title privacy | Implemented | 3 modes: full (scrubbed), redacted, off |
| Code signing | **Not yet** | Planned before production deployment |
| Auto-update | **Not yet** | Use MDM for updates |
| SAST/DAST scanning | **Not yet** | Recommended before production |
| Real-time alerts | **Not yet** | Audit log ready for SIEM integration |
| Data export controls | **N/A** | No export feature exists |
